# Security Policy

## Supported Scope

Security reports are welcome for the public code in this repository and the public release artifacts built from it.

Out of scope for this repo:

- private forks or unpublished internal deployments
- secrets that were already exposed outside this repository
- reports that require maintainers to inspect private code or private infrastructure that is not shared safely

## Report Safely

This repository does not promise a dedicated private security inbox or other private support infrastructure.

- If GitHub private vulnerability reporting is enabled for this repository, use it.
- If it is not enabled, do not post sensitive exploit details, secrets, private code, or customer data in a public issue.
- For low-risk security hardening issues that can be described safely in public, open a normal GitHub issue.
- For issues that cannot be shared safely in public, open a minimal public issue requesting maintainer contact without including the sensitive details.

## What To Include

- affected version, commit, or release artifact
- impact summary
- attack prerequisites
- reproduction steps using public-safe inputs
- any mitigations or workarounds you already verified

## Handling Expectations

- Reports are handled on a best-effort basis.
- There is no guaranteed SLA.
- Maintainers may ask for a smaller public-safe reproduction before taking action.
- Please give maintainers reasonable time to assess a report before publishing full details.
